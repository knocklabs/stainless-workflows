# Working across SDKs

_Run lifecycle tasks and shell commands across all SDK repos in your workspace using stlc test, stlc lint, stlc format, and stlc exec._

A Stainless workspace generates one SDK per target language, and each SDK lives in its own git repo on disk. As the number of targets grows, running tests, checking diffs, and committing changes across every repo becomes tedious. `stlc test`, `stlc lint`, `stlc format`, and `stlc exec` solve this by executing tasks and commands across all (or a subset of) your SDK repos from the workspace root.

## Workspace layout

After running `stlc init` and `stlc build`, a typical workspace looks like this:

```
stainless/
├── workspace.json
├── stainless.yml
└── sdks/
    ├── acme-python/
    │   └── scripts/
    │       ├── test
    │       ├── lint
    │       └── build
    ├── acme-node/
    │   └── scripts/
    │       ├── test
    │       ├── lint
    │       └── build
    ├── acme-go/
    │   └── scripts/
    │       ├── test
    │       ├── lint
    │       └── build
    └── acme-java/
        └── scripts/
            ├── test
            ├── lint
            └── build
```

Each SDK repo contains a `scripts/` directory with standard lifecycle scripts such as `test`, `lint`, and `build`. These scripts are generated by Stainless and handle language-specific tooling (pytest for Python, vitest for Node, `go test` for Go, and so on).

## Lifecycle commands: `test`, `lint`, `format`

`stlc test`, `stlc lint`, and `stlc format` invoke the corresponding `./scripts/` task across your SDK targets. They are the recommended way to run tests, linters, and formatters; for other lifecycle tasks (for example, `build`), use [`stlc exec`](#stlc-exec-arbitrary-commands).


### Run a task on specific targets

Use `--targets` (or `-t`) with a comma-separated list to limit which SDKs are included:

```sh
stlc lint --targets python,typescript
```

### Missing SDK repo or script

If a target's SDK repo has not been generated yet, the target is reported as `directory not found` and counts as a failure (see [Edge cases](#edge-cases)). If the SDK repo exists but the script does not, the underlying spawn fails and the target is reported as failed.

## `stlc exec`: arbitrary commands

`stlc exec` runs any shell command in each SDK repo. Everything after `--` is the command to execute. This is particularly useful for git operations that need to happen across all repos.

### See what changed after generation

`stlc build` commits the generated code on each SDK branch, so use `git diff --stat HEAD~1` to inspect what the latest build introduced across all SDK repos:

```sh
stlc exec -- git diff --stat HEAD~1
```

If a target's branch has no upstream yet, `git push` exits with `fatal: The current branch <branch> has no upstream branch.` for just that target; other targets push as usual. Re-run with `git push --set-upstream origin <branch>` for the failing target.

### Single-target mode

When you pass a single target with `--targets`, the header and summary are minimal. It behaves as if you were running the command directly in that directory:

```sh
stlc exec --targets go -- git diff HEAD~1
```

## A typical workflow

After changing your OpenAPI spec, a full iteration cycle combines `build`, `exec`, and the lifecycle commands:

```sh
# 1. Regenerate all SDKs (commits land on each SDK branch automatically)
stlc build --commit "feat: add payments endpoint"

# 2. Quick scan: what did this build change?
stlc exec -- git diff --stat HEAD~1

# 3. Do tests still pass?
stlc test

# 4. Lint before pushing
stlc lint

# 5. Push the build commits across SDK repos
stlc exec -- git push

# 6. Separately, commit and push tracking files from the config repo
#    so CI and collaborators can see the sealed custom code.
git add stainless/custom-code/ && git commit -m "chore: seal custom code" && git push
```

You can also run tests during a build by passing `--test` to `stlc build`.

Use `test`, `lint`, and `format` for SDK-specific lifecycle tasks and `exec` for commands that are the same everywhere (git operations, file inspection, or other lifecycle scripts).

## JSON output for CI

All lifecycle and `exec` commands accept `--format json` (or `-f json`) for machine-readable output, which is useful in CI pipelines:

```sh
stlc test --format json
```

The result is an object keyed by target identifier. Each value is the combined stdout/stderr from that target, or an error string (for example `directory not found`) when the target could not run:

```json
{
  "python": "==> Running tests with Pydantic v2\n====================== 468 passed, 1482 skipped in 3.30s =======================",
  "typescript": "==> Running tests\nTest Suites: 22 skipped, 7 passed, 7 of 29 total\nTests:       227 skipped, 216 passed, 443 total\nTime:        2.893 s",
  "go": "FAIL\tgithub.com/acme/acme-go\t0.847s"
}
```

The process exit code is non-zero if any target failed.

## Edge cases

### Target directory does not exist

If a target is configured in `workspace.json` but its repo has not been generated yet, `stlc test` / `lint` / `format` reports an error with a helpful message:

```
Running `./scripts/test` in 2 targets: python, ruby

✱ Python ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

==> Running tests with Pydantic v2
====================== 468 passed, 1482 skipped in 3.30s =======================

✱ Ruby ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✗ directory not found
```

Run `stlc build --targets ruby` to create the missing SDK before retrying.

### Target lacks the requested script

If the SDK repo exists but `./scripts/<task>` is missing, the underlying spawn fails (typically with `ENOENT`). The target is reported as failed and the process exit code is non-zero.
